# Multi-Provider Token Burn Design

## Goal

Add a new **Burn** tab that visualizes daily token usage across local AI
providers. The first version supports Claude Code and Codex, while defining a
provider-neutral data model that can support ChatGPT or other sources later.

The Burn tab complements the existing Claude-focused analytics. It does not
replace or alter the current Overview, Prompts, Sessions, Projects, Skills,
Tips, or Settings behavior.

## User Experience

The new Burn tab is inspired by a calendar-style token-burn dashboard and
contains:

- Range controls: `90d`, `180d`, `1y`, and `All`
- A metric toggle:
  - **Workload**: all model-processed tokens
  - **Billable**: provider-normalized billable-equivalent tokens
- Summary cards for total usage, day-to-date usage, and peak day
- A weekly-total trend line
- Calendar heatmap lanes for:
  - Total
  - Codex
  - Claude
  - Any future provider with available data
- Provider totals with exact or estimated status labels
- A peak-day table showing each provider's contribution

Workload is the default metric because it best represents total AI activity.

## Architecture

### Provider-Neutral Daily Usage

Add a dedicated daily aggregation table rather than forcing Codex records into
the existing Claude-specific `messages` table.

Each row represents one provider on one local calendar day:

```text
provider
day
input_tokens
output_tokens
cached_input_tokens
cache_create_tokens
reasoning_output_tokens
workload_tokens
billable_tokens
accuracy
updated_at
```

The `(provider, day)` pair is unique.

`accuracy` is a small explicit status such as `exact` or `estimated`. Claude
Code and Codex are `exact` because their local logs contain token counters.
Future sources without exact counters can be marked `estimated`.

### Metric Definitions

The provider adapters normalize source-specific usage into common fields.

**Workload tokens**

All tokens processed or generated by the model:

```text
input + output + cache create + reasoning output
```

Where a provider reports cached input as a subset of input, cached input is not
added again. Provider adapters are responsible for avoiding double counting.

**Billable tokens**

A provider-normalized usage count intended for relative comparison:

```text
non-cached input + output + cache create + reasoning output
```

This is not a currency estimate. Provider pricing and subscription accounting
are not sufficiently comparable to present one universal dollar value on the
Burn tab.

### Claude Adapter

Reuse the already-scanned Claude `messages` table as the source of truth.
Aggregate messages by local calendar day into the new daily usage table.

Claude's cache-read and cache-create counters are mapped according to the
existing dashboard semantics. The adapter must preserve existing Claude totals
and must not modify the current message scanner behavior.

### Codex Adapter

Scan `~/.codex/sessions/**/*.jsonl`.

Codex emits cumulative `token_count` events containing:

- `input_tokens`
- `cached_input_tokens`
- `output_tokens`
- `reasoning_output_tokens`
- `total_tokens`

For each session, use the final valid cumulative token-count event as the
session total. This avoids summing cumulative snapshots and over-counting.
Attribute the session total to the local calendar day of the final event.

This first version intentionally aggregates Codex at session/day granularity.
Per-prompt Codex analytics are outside scope.

### Incremental Refresh

Track scanned Codex files using the existing `files` mechanism or an equivalent
provider-aware file-state table. A changed Codex session must replace its prior
daily contribution rather than add a second copy.

After each regular scan:

1. Refresh changed Claude source data using the existing scanner.
2. Refresh changed Codex session totals.
3. Rebuild affected provider/day aggregate rows.
4. Publish the existing SSE scan event so the Burn tab refreshes.

## API

Add a provider-neutral endpoint:

```text
GET /api/burn?since=<iso-date>&until=<iso-date>&metric=workload|billable
```

The response contains:

- Selected range and metric
- Overall total
- Day-to-date total
- Peak day and value
- Daily totals by provider
- Weekly totals
- Provider totals and accuracy status
- Ranked peak days with provider contributions

The server validates `metric`; unsupported values fall back to `workload`.

## Frontend

Add `/burn` to the existing hash router and top navigation.

Implement `web/routes/burn.js` as a focused route. Reuse existing formatting
helpers and ECharts where practical. The calendar heatmap may use a custom
CSS-grid renderer because multiple aligned provider lanes are easier to control
that way than with separate ECharts calendars.

Heatmap cells use logarithmic color bins so unusually large Codex context
events do not flatten normal daily variation. Tooltips show the exact date,
provider, selected metric, and token count.

The page must remain useful when:

- Only Claude data exists
- Only Codex data exists
- A provider has no data in the selected range
- The selected range has no usage

## Error Handling

- Missing provider source directories are treated as unavailable, not errors.
- Malformed or partial JSONL lines are skipped and retried after the file
  changes.
- A Codex session without a valid cumulative token-count event contributes no
  usage.
- Provider scan errors are isolated so one provider cannot prevent other data
  from refreshing.
- The API returns empty arrays and zero totals for a valid range with no data.

## Privacy

The feature remains fully local. It reads token counters and timestamps from
local transcripts and serves aggregated data through the existing localhost
server. The Burn API does not expose prompt or response text.

## Testing

Add focused tests for:

- Parsing final cumulative Codex token-count events
- Avoiding cumulative snapshot double-counting
- Rescanning changed Codex sessions without duplication
- Provider metric normalization
- Claude daily aggregation preserving current totals
- Burn API range and metric validation
- Empty, malformed, partial, and missing-source cases
- Combined Claude and Codex totals

Run the complete Python unit test suite after implementation and visually
verify the Burn tab against local Claude and Codex data.

## Out of Scope

- ChatGPT ingestion in the first version
- Per-prompt or tool-level Codex analytics
- Named workstream inference
- Universal cross-provider dollar-cost estimates
- Replacing existing Claude-focused dashboard pages
